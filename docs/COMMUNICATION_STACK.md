# 통신 스택 완전 분해 — RS485 · Modbus-RTU · UART · USB는 각 계층에서 누가 책임지나

> LORDDOM LGUB 센서의 거리값이 우분투 응용 프로그램에 도착하기까지, 실제 부품과
> 소프트웨어를 계층별로 쪼갠 문서다. 핵심 목적은 "RS485와 Modbus-RTU가 서로 다른
> 계층이며 직접 만나지 않는다"는 것을 부품 단위로 이해하는 것이다.
>
> 관련 문서: [PROTOCOL.md](./PROTOCOL.md), [HOW_WE_BUILT_SDK_WITHOUT_DOCS.md](./HOW_WE_BUILT_SDK_WITHOUT_DOCS.md)

## 0. 가장 중요한 전제

**RS485와 Modbus-RTU는 서로 직접 만나지 않는다.**
둘 사이에는 반드시 **UART(바이트 ↔ 비트 직렬화)** 계층이 끼고, USB 어댑터를 쓰면
**USB** 라는 터널이 하나 더 들어간다. 이 계층들을 통과하는 내내 변하지 않고 흐르는
단 하나의 실체는 **바이트 열(byte stream)** 이다.

- **Modbus-RTU** = 바이트 열을 *만들고 해석* 하는 소프트웨어 (양쪽 끝)
- **UART** = 바이트를 *비트로 쪼개고 다시 합치는* 계층 (baud/parity/stop 소속)
- **RS485** = 그 비트를 *A/B 차동 전압으로 실어 나르는* 전기 규격 (트랜시버+케이블)
- **USB** = 우리 셋업에만 있는 *터널* (바이트 열을 포장했다 그대로 푼다)

## 1. 전체 스택 (응답 경로: 센서 → 우분투)

요청 경로(우분투 → 센서)는 정확히 이 반대 방향이다.

```
[센서 펌웨어 앱]  거리 계산 → 값 602(count)
      │  (아직 Modbus도 RS485도 아님. 그냥 숫자)
      ▼
[센서 Modbus 슬레이브 스택]  값을 Input Reg 0x0002에 담아 프레임 조립
      │   01 04 02 02 5A 38 6B   ← 바이트 열 완성 (여기까지 Modbus-RTU)
      ▼
[센서 MCU UART 송신기]  각 바이트를 start+8bit+stop 으로 직렬화 (9600bps, TTL 전압)
      │   비트의 연속 (아직 A/B 아님, 그냥 0/3.3V 로직)
      ▼
[센서 RS485 트랜시버 칩]  로직 비트 → A/B 차동 전압 + 송신방향 제어   ← RS485
      │
      ▼
[M12 케이블 트위스트 페어 A/B]  물리적 전선으로 차동 신호 전송        ← RS485 물리 매체
      │
      ▼
[어댑터 RS485 트랜시버 칩]  A/B 차동 전압 → 로직 비트로 복원          ← RS485
      │
      ▼
[어댑터 FT232 UART 수신기]  비트열 → 바이트 역직렬화 (9600 설정 기준)
      │   바이트 열 복원됨 (01 04 02 ...)
      ▼
[어댑터 FT232 USB 장치부]  바이트를 USB Bulk 패킷에 실어 전송         ← USB (RS485/Modbus 무관)
      │   D+/D- 차동 USB 신호
      ▼
[USB 케이블 + 우분투 xHCI 호스트 컨트롤러(HW)]  USB 패킷 수신
      │
      ▼
[리눅스 커널: usbcore + ftdi_sio 드라이버]  USB 패킷 → 바이트 열, /dev/ttyUSB0 노출
      │  (termios 설정을 FT232에 control transfer로 내려 baud 프로그래밍하는 것도 이 계층)
      ▼
[tty 계층 / termios (raw 모드)]  바이트를 가공 없이 유저스페이스로 전달
      │
      ▼
[우리 SerialPort::read]  raw 바이트 수신 (01 04 02 02 5A 38 6B)
      ▼
[우리 ModbusRtu 파서]  프레임 검증 + CRC 확인 → 레지스터값 602 추출  ← Modbus-RTU (마스터측)
      ▼
[우리 LgubSensor SDK]  602 × 0.17mm = 102.3mm → 0.102 m             ← 의미 부여 (표준 아님, 역설계)
      ▼
[응용 프로그램]  거리 = 0.102 m 사용
```

## 2. 계층별 책임 주체

| # | 계층 역할 | 실제 책임 주체 | 소속 |
|---|---|---|---|
| 1 | 물리량 측정, 값 생성 | 센서 **펌웨어 앱 로직** | 제조사 (표준 아님) |
| 2 | "값을 어느 레지스터·어느 기능코드로" 결정 | 센서 **Modbus 슬레이브** / 우리 마스터 | **Modbus (앱)** |
| 3 | 프레임 조립 `[주소][기능][데이터][CRC]`, RTU 프레임 경계(3.5문자 침묵) | 같은 Modbus 스택 (양쪽) | **Modbus-RTU (데이터링크)** |
| 4 | 바이트 ↔ 비트 직렬화 (start/stop/parity, baud) | 센서 **UART** / 어댑터 **FT232 UART부** | **UART** (RS485도 Modbus도 아님) |
| 5 | 로직 비트 ↔ **A/B 차동 전압**, 방향 제어 | **RS485 트랜시버 칩** (센서·어댑터 각 1개) | **RS485 (물리 전기)** |
| 5b | 차동 신호 물리 전송 | **트위스트 페어 케이블(A/B)** | RS485 물리 매체 |
| 6 | 바이트 ↔ USB 패킷 (터널링) | 어댑터 **FT232 USB부** | **USB** |
| 7 | USB 전송/수신 HW | 우분투 **xHCI 호스트 컨트롤러** | USB |
| 8 | USB→바이트열, `/dev/ttyUSB0` 생성, baud 프로그래밍 | 리눅스 커널 **`ftdi_sio`** | OS |
| 9 | raw 바이트 유저스페이스 전달 | **tty / termios** | OS |
| 10 | 프레임 파싱 + CRC 검증 | 우리 **ModbusRtu 코드** | **Modbus-RTU (마스터측)** |
| 11 | raw → 물리단위(×0.17mm) | 우리 **LgubSensor SDK** | 역설계 지식 (표준 아님) |

## 3. 반드시 가져갈 4가지

**① RS485는 오직 5번(트랜시버+케이블)뿐이다.**
RS485는 "비트 하나를 A선과 B선의 **전압 차이**로 표현하는 전기 규격"일 뿐이다.
프레임·바이트·Modbus가 뭔지 전혀 모른다. A/B에는 "프레임"이 아니라, UART가 이미
잘게 쪼갠 **비트(전압 펄스)** 가 흐른다. 반이중이라 **"지금 누가 말할 차례인지"(방향
전환 DE/RE)** 도 트랜시버가 책임진다.

**② Modbus-RTU는 오직 2·3·10번(순수 소프트웨어)뿐이다.**
Modbus는 "바이트를 어떻게 배열해 의미를 만들지"만 정한다. A선/B선·전압은 전혀
모른다. Modbus 입장에서 아래는 그냥 "바이트를 넣으면 전달되는 시리얼 선"이다.
그래서 Modbus-RTU는 RS485 위에서도, RS232 위에서도, USB 터널 위에서도 똑같이 돈다.

**③ 둘을 잇는 진짜 다리는 4번 UART다.**
RS485도 Modbus도 정의하지 않지만 없으면 통신이 안 되는 계층. **baud(9600)·패리티·stop
bit가 여기 소속.** "9600 8N1"을 맞춰야 했던 이유는 Modbus/RS485 문제가 아니라
**UART 정합** 문제였다.

**④ USB(6·7·8번)는 우리 셋업에만 있는 터널이다.**
컴퓨터에 RS485 포트가 직접 있었다면 USB 계층 전체가 없다. FT232+`ftdi_sio`는
바이트 열을 USB로 포장했다 그대로 푸는 역할만 하며, 흐르는 바이트는 조금도 안 바뀐다.

## 4. OSI 대응 (근사)

실제 임베디드 시리얼 스택은 OSI처럼 깔끔히 나뉘지 않는다. 직관용 근사 대응:

| OSI 계층 | 이 시스템에서 | 담당 |
|---|---|---|
| L7 응용 | 거리값 의미(레지스터맵·단위) | 펌웨어 앱 / 우리 SDK |
| L2~L7 걸침 | Modbus-RTU (프레임+기능코드+CRC) | Modbus 스택 (양끝) |
| L2 데이터링크 | UART 바이트 프레이밍 (start/stop/parity) | UART 주변장치 |
| L1 물리 | RS485 차동 전기 + 케이블 | 트랜시버 + 트위스트 페어 |
| (별도 스택) | USB 터널 | FT232 + 커널 드라이버 |

## 5. 한 줄 결론

> **바이트 열**이 전 계층을 관통하는 불변의 실체다. **Modbus-RTU**(SW)가 바이트 열을
> 만들고 해석하고, **UART**가 그 바이트를 비트로 쪼개고 합치며, **RS485**(트랜시버+케이블)가
> 그 비트를 A/B 전압으로 실어 나른다. Modbus와 RS485는 서로를 모르며, 오직
> "바이트 → 비트 → 전압"의 계단으로만 연결된다.
