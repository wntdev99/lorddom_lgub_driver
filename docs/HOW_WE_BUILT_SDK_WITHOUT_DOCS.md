# SDK가 없는데 어떻게 SDK를 만들 수 있었나

> LORDDOM LGUB 센서는 제조사가 SDK도 프로토콜 문서도 공개하지 않는다.
> 그런데도 동작하는 C++ SDK를 만들 수 있었다. 이 문서는 "왜 가능했고,
> 무엇을 어떤 순서로 했는지"를 처음 보는 사람도 이해할 수 있게 기록한다.
>
> 결과물: [PROTOCOL.md](./PROTOCOL.md)(역설계 명세), 상위 [README](../README.md),
> `include/lorddom_lgub/`(SDK), `tools/lgub_probe.cpp`(프로빙 도구).

---

## 0. 먼저 오해부터 풀기 — "SDK 없음"의 진짜 의미

"SDK가 없다"는 말은 무섭게 들리지만, 실제로 없던 건 **아주 작은 부분**이었다.
센서와 대화하는 일은 딱 3개 층으로 나뉜다.

| 층 | 무엇인가 | 우리 경우 | 공개 여부 |
|---|---|---|---|
| ① 전선 | 어떤 전기 신호로 연결하나 | **RS485** | 공개 표준 |
| ② 언어 | 그 신호 위에서 무슨 규칙으로 말하나 | **Modbus-RTU** | 공개 표준 |
| ③ 사전 | 주고받는 숫자가 뭘 뜻하나 | 레지스터 `0x0002`=거리, ×0.17mm | **이것만 비공개** |

핵심: **①전선과 ②언어는 전 세계가 쓰는 공개 표준**이다. 누구나 구현할 수 있다.
"SDK가 없다"가 가린 건 **③사전(어느 숫자가 거리인지)** 하나뿐이었고,
그마저도 Modbus 응답이 **자기 자신을 설명하는(self-describing)** 구조라
실물을 두드려보면(probing) 알아낼 수 있다.

> **비유**: 모르는 나라에 떨어졌지만 그 나라가 *한국어를 쓴다*는 걸 알면,
> 사전이 없어도 대화가 된다. 모르는 건 "그 사람 이름" 정도고, 그건 물어보면 된다.
> 우리가 한 게 딱 그것이다.

## 1. 언어의 종류부터 확정 (RS485 + Modbus)

가장 먼저 "이 센서가 무슨 언어를 쓰나"를 확정해야 했다. `docs/`의 데이터시트
이미지에서 **"Digital output: RS485, Modbus communication protocol"** 문구와
배선도를 찾았다. → ①전선=RS485, ②언어=Modbus 확정.
여기서부터는 **표준 문서가 존재하는 세계**로 들어온다.

## 2. 물리적으로 연결 (전선 잇기)

데이터시트 M12 5핀 배선도: 노랑=A(+), 초록=B(−), 빨강=전원+(10~30V), 검정=GND.
이때 **"RS485는 별도 전원이 필요"**임을 확인 — 이게 없으면 아무 응답도 오지 않는다.

## 3. 언어를 코드로 직접 구현 (Modbus-RTU 마스터)

Modbus는 공개 표준이라 규칙이 명확하다. 그 규칙대로 코드를 작성:
- **프레임 조립**: `[주소][기능코드][레지스터][개수][CRC]`
- **CRC16**: 프레임 무결성 체크섬(표준 알고리즘, 다항식 0xA001)
- 응답 파싱 / 예외 응답 처리 / 타임아웃

이것이 SDK의 뼈대다. 이 시점에 센서 없이도 "말할 준비"가 끝난다.

## 4. 하드웨어 없이 먼저 검증 (증거 기반)

센서를 연결하기 **전에**, 내가 짠 Modbus 코드가 규격에 맞는지 증명했다.
Modbus 표준의 **알려진 정답 벡터**(`01 03 00 00 00 01` → CRC `84 0A`)로
오프라인 단위테스트를 통과시켰다.
→ "센서가 무응답이면 내 코드 탓인가?"라는 변수를 미리 제거.
(참고: `test/test_crc16.cpp`)

## 5. 통신 파라미터 찾기 (누구에게, 얼마나 빨리 말하나)

어댑터 연결 후에도 모르는 것: **슬레이브 주소**와 **보드레이트**.
가장 흔한 조합(9600, 주소 1)을 먼저 두드리자 **응답이 왔다.** → 통신 성립.
(실패했다면 `lgub_probe scan`으로 주소 1~247 × 보드레이트 전수조사할 준비도 돼 있었다.)

## 6. 데이터가 어디 있나 (레지스터 스캔)

두 종류 읽기(`0x03` Holding, `0x04` Input)로 `0x0000~0x000F`를 훑었다.
- `0x03`: 대부분 0, `0x0002`만 3 → **설정값들**(거리 아님)
- `0x04`: `0x0000~0x0003`만 살아있는 값 → **여기가 실시간 데이터**

## 7. 그중 뭐가 "거리"인가 (움직이며 관찰)

물체를 **원거리 → 100mm 근처**로 옮기며 값을 관찰:
- `0x0000`, `0x0002`는 거리 따라 **같이 변함** → 거리 관련
- `0x0001`은 고정 → 상태값
- 두 값의 비가 항상 1.709로 일정 → **같은 거리를 단위만 다르게 표현**

## 8. 숫자를 mm로 (배율 역산 + 데이터시트 대조)

`약 100mm`에서 `0x0002 = 602~621`. `602 × 0.17 = 102mm` → 딱 맞음.
그리고 이 **0.17mm가 데이터시트의 "분해능 0.17mm"와 정확히 일치** →
우연이 아니라 제조사 설계값임을 확정.
③사전 완성: **`0x0002` = 거리, 1 count = 0.17mm.**

## 9. 고수준 API로 포장 = SDK 완성

위 모든 걸 `sensor.read_distance()` 한 줄로 감쌌다. 사용자는 레지스터도 CRC도
몰라도 된다. **이 "포장"이 바로 SDK다.** 실측에서 `0.102 m`가 나오며 전 과정이 닫혔다.

## 한 문장 요약

> SDK가 없어도 됐던 이유는, 센서의 언어(RS485+Modbus)가 **공개 표준**이라
> 내가 직접 구현할 수 있었고, 비공개였던 "어느 숫자가 거리인가"는 실물을
> **두드려서(probing) 역산**했기 때문이다.

## 일반화 — 문서 없는 다른 센서에도 통하는 절차

1. **언어 확정**: 인터페이스가 RS485/Modbus/UART/CAN/I2C 중 무엇인지 (라벨·데이터시트·검색)
2. **물리 연결**: 배선·전원·종단 확인 (여기서 대부분의 "무응답"이 발생)
3. **표준 계층 구현 + 오프라인 검증**: 프로토콜은 표준이므로 문서가 있다. 알려진 벡터로 먼저 검증
4. **파라미터 탐색**: 주소·속도 스캔으로 통신 성립
5. **데이터 위치 탐색**: 레지스터/메시지 스캔
6. **의미 부여**: 물리량을 바꿔가며 관찰해 어느 값이 무엇인지 역산
7. **배율 확정**: 알려진 기준값 대조 (+ 데이터시트 스펙 교차검증)
8. **API로 포장**: 위 지식을 한 줄 호출로 캡슐화 → SDK
